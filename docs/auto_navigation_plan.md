# 自动导航路线和执行方案

## 目标

第一版自动导航用于低速航点跟随：用户在 Android App 地图上维护路线，选择路线后由 App 根据实时 GPS 位置和手机指南针航向计算左右推进输出，并持续发送给 ESP32。ESP32 不保存路线、不独立规划路径，只执行 App 发来的左右功率命令，并保留未解锁、失联、低压、过流、过温和故障回空闲保护。

自动导航只能作为低速辅助功能，不替代人工观察、人工接管和硬件急停。

## 安全边界

- 自动导航不得自动绕过上电默认锁定；执行前必须由用户在 App 中明确确认。
- App 必须持续发送控制心跳；ESP32 收不到心跳时回空闲油门。
- Android 手机必须固定在桨板上，且手机朝向相对船体不变；手机指南针失效或超过 `1.5s` 无更新时，App 必须停止自动导航并回空闲。
- GPS 未定位、卫星数不足、坐标缺失、路线少于 2 点、距离目标异常跳变时，App 必须拒绝执行或停止执行。
- 自动导航第一版默认低速，建议基础推进档位限制在 `10%..35%`，转向修正可配置但不得超过当前 App/ESP32 的最终限幅。
- 手动停止按钮必须始终可见，并立即发送 `ARM=0;L=0;R=0` 或至少 `L=0;R=0` 的保守命令。
- 手动遥控、有线控制器、急停和 ESP32 保护优先级高于自动导航。

推荐控制优先级：

```text
硬件急停 / 断电 > 有线控制器 > ESP-NOW 遥控器 > Android 手动控制 > Android 自动导航
```

## Android 功能

当前 Android Debug 版已实现第一版 App 端路线功能：

- 路线保存在手机本地 `files/navigation_routes/routes.json`。
- 导航页“路线”入口可添加、编辑、删除和执行路线。
- 路线编辑模式支持地图点击添加航点、撤销最后航点、保存和取消。
- 自动导航执行时显示当前速度、目标航点、到目标距离、航向误差、当前档位、左右输出、加减档和停止按钮。
- 自动导航仍复用现有 App 控制心跳，ESP32 不保存路线、不独立规划。

### 路线入口

导航页新增“路线”按钮，点击后展开路线列表。

路线列表显示：

- 路线名称。
- 航点数量。
- 估算总距离。
- 最近修改时间。
- 操作：执行、编辑、删除。

### 路线编辑模式

新增路线后进入路线编辑模式：

- 地图点击添加航点。
- 航点按添加顺序连成路线。
- 航点可选中后删除。
- 可保存、取消。
- 保存前至少需要 `2` 个航点。

第一版可以先不做拖动重排；如果需要调整路线，先删除点并重新添加。后续再增加拖动点、插入点、重命名和路线复制。

### 执行模式

用户选中路线后点击“执行”，进入自动驾驶界面：

- 地图显示路线、当前目标航点和当前位置。
- 显示当前速度。
- 显示目标航点编号、到目标距离、航向误差。
- 显示当前自动导航档位。
- 提供加档、减档和停止按钮。

执行前检查：

- 已连接 ESP32。
- GPS `GPS_FIX=1`。
- 卫星数满足最低要求，建议 `GPS_SAT >= 4`。
- 当前实时坐标可用。
- 手机航向可用。
- 路线至少 `2` 个点。

## 路线数据

Android 本地保存路线，不写入 ESP32。

建议存储位置：

```text
files/navigation_routes/routes.json
```

建议数据结构：

```json
{
  "routes": [
    {
      "id": "route-20260617-001",
      "name": "深圳湾测试路线",
      "created_at": 1781656200,
      "updated_at": 1781656500,
      "points": [
        {"lat": 22.5206, "lon": 113.9438},
        {"lat": 22.5212, "lon": 113.9445}
      ]
    }
  ]
}
```

## 控制策略

第一版使用“指向目标航点”的简单闭环：

1. 从路线中取当前目标航点。
2. 对自动导航使用的 GPS 坐标做轻量低通滤波；如果坐标跳变超过异常阈值，重置滤波点。
3. 用滤波后的 GPS 坐标计算目标方位 `target_bearing` 和目标距离 `distance_m`。
4. 接近航点时不再只追“当前位置到航点”的瞬时方位，而改用后续路线段/下一个航点作为稳定引导方向，避免 GPS 抖动在最后几米把方位角放大成大幅跳变。
5. 用手机指南针航向 `heading` 计算航向误差 `error = shortest(target_bearing - heading)`。
6. 基础推进由自动导航档位给出。
7. 根据航向误差计算左右差速修正。
8. 到达航点半径内后切到下一个航点。
9. 到达最后航点后自动停止。

建议初始参数：

| 参数 | 初始值 | 说明 |
| --- | --- | --- |
| 到达半径 | `8m` | GPS 低速漂移较大，半径不宜太小 |
| 接近航点稳定引导半径 | `18m` | 进入该范围后参考后续路线方向，减少方位角抖动 |
| GPS 滤波系数 | `0.35` | 新坐标权重，降低低速漂移对控制的影响 |
| GPS 跳变重置阈值 | 默认 `5m`，设置范围 `3m..30m` | 超过该距离视为异常跳变或重定位，重置滤波 |
| 基础档位 | `15% / 22% / 30%` | App 自动导航档位 |
| 最大修正 | `25%` | 左右差速最大修正 |
| 满修正角度 | `45°` | 大误差时达到最大修正 |
| GPS 最低卫星数 | `4` | 低于该值不执行 |
| 航向超时 | `1.5s` | 手机指南针无更新则退出 |
| 坐标超时 | `3s` | GPS 坐标无更新则退出 |

差速示例：

```text
base = 当前档位基础推进百分比
correction = clamp(error / full_error, -1, 1) * max_correction

left = base + correction
right = base - correction
```

如果误差过大，例如 `abs(error) > 120°`，第一版建议先降速或停止转向，不建议大功率原地追航向。

## 航迹线锁定第一阶段

航迹线锁定是单段路线跟踪模式，用于水流或侧风造成“板头方向正确但 GPS 轨迹逐渐偏离”的场景。第一阶段只在 Android App 中实现，不新增 ESP32 路线协议；ESP32 仍只接收 App 计算后的左右推进功率，并保留失联、限幅、低压、过流、过温和故障保护。

启动时 App 记录：

- `origin`：当前 GPS 位置。
- `line_bearing`：当前所选航向源读数，作为目标航迹线方向。
- `gearIndex`：沿用自动导航低速档位，初始为第一档。

每个控制心跳周期：

1. 读取并滤波当前 GPS 点；GPS 跳变超过设置页阈值时重置滤波。
2. 计算当前点到目标航迹线的有符号横向偏差 `cross_track_error_m`，右侧为正、左侧为负。
3. 使用 LOS 引导计算纠偏角：

```text
correction_angle = atan2(-cross_track_error_m, lookahead_m)
target_heading = line_bearing + clamp(correction_angle, -max_angle, +max_angle)
```

4. 用所选航向源计算 `heading_error = shortest(target_heading - heading)`。
5. 沿用自动导航的低速基础档位和差速修正，输出最终左右功率。

第一阶段参数：

| 参数 | 初始值 | 说明 |
| --- | --- | --- |
| 前视距离 `lookahead_m` | `12m` | 横向偏差转为航向修正的尺度 |
| 最大纠偏角 | `25°` | 防止 GPS 抖动导致大角度转向 |
| 最大横向偏差 | `15m` | 超出后停止航迹线锁定并回空闲 |
| 基础档位 | `15% / 22% / 30%` | 复用自动导航档位 |
| 最大差速修正 | `25%` | 复用自动导航修正上限 |
| GPS 最低卫星数 | `4` | 低于该值不执行 |
| 航向超时 | 手机 `1.5s`，ESP32 IMU 测试模式 `500ms` | 复用现有航向源约束 |

主控九轴 IMU 上报的 `YBY` 是模块原始 yaw。第一阶段在 Android App 中按指南针航向约定转换为 `heading = normalize(360° - YBY)` 后再用于地图箭头、航迹线锁定和航向误差计算；设置页保留原始 `YBY` 读数用于排查。

安全边界：

- 航迹线锁定不得自动解锁；启动前必须已连接、已手动解锁、GPS 定位、卫星数满足要求、航向源有效。
- 用户停止、GPS 无效、航向无效、横向偏差过大或目标航向误差超过 `120°` 时，App 必须停止推进或锁定回空闲。
- 该模式只能作为低速辅助；测试顺序仍按岸上模拟、岸上低功率、静水低速短时、逐步扩大距离执行。

## App 到 ESP32 控制

第一版复用现有 App 控制命令心跳，不新增 ESP32 路线协议。App 在自动导航执行中周期性发送最终左右功率：

```text
SRC=APP;ARM=1;L=<left>;R=<right>;H_SRC=PHONE
```

停止时发送：

```text
SRC=APP;ARM=0;L=0;R=0;H_SRC=PHONE
```

ESP32 侧仍按现有规则处理：

- 未解锁或故障时不输出推进。
- 控制心跳超时回空闲。
- 输出受限幅、斜率、低压/过流/过温保护约束。

后续如果要让 ESP32 独立执行路线，必须另起设计：路线校验、持久化、航点状态机、独立 GPS/IMU 闭环、人工接管和安全审查都需要在 ESP32 侧实现。

## 测试阶段

1. App 纯 UI 测试：添加、删除、保存路线。
2. 回放/模拟位置测试：用模拟 GPS 点验证航点切换和左右输出计算，但不连接推进。
3. 岸上空载测试：ESP32 接收自动导航命令，但 ESC 输出保持空闲或断开电机。
4. 岸上低功率单电机测试。
5. 岸上低功率双电机测试。
6. 静水低速短时测试，人工随时接管。
7. 逐步扩大路线距离和档位。

任何阶段出现 GPS 漂移、手机方向不稳定、通信延迟、意外转向或停止不及时，都必须退回上一阶段。
